# Requirement Traceability Matrix
## University Management System (ICT Education)

**Source inputs:** `docs/product_proposal.pdf`, `docs/Requirement_Analysis.md`, `docs/System_Architecture.md`, `docs/Database_Design.md`, `docs/Implementation_Roadmap.md`
**Scope:** Traces every Functional Requirement (FR-001‚ÄìFR-056) and Non-Functional Requirement (NFR-001‚ÄìNFR-016) from `Requirement_Analysis.md` through to database tables, backend APIs, frontend pages, and applicable validation rules.

**Priority scale:**
- **Critical** ‚Äî core grading-rubric feature (API/Web App/DB per proposal ¬ß2); system cannot function or pass core grading without it.
- **High** ‚Äî required feature explicitly specified in the proposal, non-optional module.
- **Medium** ‚Äî required but lower-impact, or part of the "(Optional)" Fees module (still specified in full ‚Äî see A-005).
- **Low** ‚Äî supporting/cross-cutting requirement not independently gradable but needed for correctness.

**Status columns** are all initialized to **Pending** per instruction, to be updated as implementation proceeds through `Implementation_Roadmap.md` milestones.

## Known Gaps Affecting Traceability

The following items are cross-referenced from `Requirement_Analysis.md` ¬ß14 because they directly affect the "Backend API" or "Frontend Pages" columns above and could not be resolved without inventing scope:

- **FR-052, FR-053, NFR-009**: No notification-management endpoints (`GET /notifications`, `PUT /notifications/{id}/read`) are listed in the proposal's ¬ß6 API spec, despite being required by the corresponding features. Marked with a gap note rather than a fabricated endpoint; `Implementation_Roadmap.md` Milestone 9 proposes the missing endpoints as an implementation necessity.
- **FR-050**: No schedule-change-request endpoint is listed in the proposal's ¬ß6 API spec. Marked with a gap note rather than a fabricated endpoint; `Implementation_Roadmap.md` **Milestone 4** (not Milestone 9 ‚Äî corrected during the M4 pre-implementation review; this line previously grouped FR-050 with the Milestone 9 notification gaps above, which was inconsistent with `API_Contract.md` ¬ß7.6-7.7's own text stating both endpoints are "implemented per Implementation_Roadmap.md Milestone 4," and with `schedule_change_request.py` being an M4 file) proposes `POST /schedule/change-requests` and `POST /schedule/change-requests/{id}/resolve` as an implementation necessity.
- **FR-046‚ÄìFR-049**: No dedicated "Admin schedule management" screen is named in `Requirement_Analysis.md` ¬ß7; scheduling admin actions are assumed to live within the Timetable page's admin-mode, consistent with how Section 7 does not enumerate a separate screen for it.
- **`class_session`/`enrollment` creation (no FR number ‚Äî not a proposal requirement):** `POST /schedule/class-sessions` and `POST /schedule/enrollments` (Milestone 4) are Derived Engineering Additions, not traceable to any proposal FR ‚Äî both `class_session` and `enrollment` are referenced as foreign keys throughout the Scheduling/Exams/Attendance endpoints above but had no creation endpoint anywhere in the source documents, and without them `POST /schedule` (FR-046) cannot be exercised. Confirmed with the user during the Milestone 4 pre-implementation review; see `Proposal_vs_Engineering_Additions.md` for the full classification.
- **FR-032, FR-037, FR-038, FR-041**: Parent-facing data access reuses existing role-scoped endpoints in a parent-scoped mode (per `System_Architecture.md` ¬ß6), since the proposal does not define separate Parent-only endpoints for attendance/results ‚Äî only fees has explicit Parent access in the Access column (`Requirement_Analysis.md` ¬ß8). **Note (corrected during the Milestone 5 pre-implementation review):** this is not uniformly "the Student-facing endpoint" ‚Äî FR-032 (attendance) specifically reuses the Teacher/Admin **class-based** endpoint (`GET /attendance/{classId}`, parent-scoped via `parent_student_link`), not `GET /attendance/me`, per that row's own Backend API column above and `API_Contract.md` ¬ß4.3. FR-037/FR-038/FR-041 (results/fees) do reuse their respective `/me` endpoints ‚Äî the pattern differs per domain, so "Student-facing" was overly broad as originally worded.
- **`class_session`/`enrollment` roster (no FR number ‚Äî not a proposal requirement):** `GET /schedule/class-sessions/{class_session_id}/roster` (Milestone 5) is a Derived Engineering Addition ‚Äî `UI_Wireframes.md` ¬ß15 (Teacher: Attendance Marker) requires the roster to load with enrolled students pre-populated before marking, but no endpoint anywhere returned "students enrolled in a class session." Confirmed with the user during the Milestone 5 pre-implementation review; see `Proposal_vs_Engineering_Additions.md` for the full classification.
