# Requirement Traceability Matrix
## University Management System (ICT Education)

**Source inputs:** `docs/product_proposal.pdf`, `docs/Requirement_Analysis.md`, `docs/System_Architecture.md`, `docs/Database_Design.md`, `docs/Implementation_Roadmap.md`
**Scope:** Traces every Functional Requirement (FR-001тАУFR-056) and Non-Functional Requirement (NFR-001тАУNFR-016) from `Requirement_Analysis.md` through to database tables, backend APIs, frontend pages, and applicable validation rules.

**Priority scale:**
- **Critical** тАФ core grading-rubric feature (API/Web App/DB per proposal ┬з2); system cannot function or pass core grading without it.
- **High** тАФ required feature explicitly specified in the proposal, non-optional module.
- **Medium** тАФ required but lower-impact, or part of the "(Optional)" Fees module (still specified in full тАФ see A-005).
- **Low** тАФ supporting/cross-cutting requirement not independently gradable but needed for correctness.

**Status columns** are all initialized to **Pending** per instruction, to be updated as implementation proceeds through `Implementation_Roadmap.md` milestones.

## Known Gaps Affecting Traceability

The following items are cross-referenced from `Requirement_Analysis.md` ┬з14 because they directly affect the "Backend API" or "Frontend Pages" columns above and could not be resolved without inventing scope:

- **FR-052, FR-053, NFR-009**: No notification-management endpoints (`GET /notifications`, `PUT /notifications/{id}/read`) are listed in the proposal's ┬з6 API spec, despite being required by the corresponding features. Marked with a gap note rather than a fabricated endpoint; `Implementation_Roadmap.md` Milestone 9 proposes the missing endpoints as an implementation necessity.
- **FR-051 recipients correction (resolved during the Milestone 9 pre-implementation review):** `Requirement_Analysis.md`'s original wording ("notifications to students") and this document's prior row both said Student only тАФ `UI_Wireframes.md` ┬з16's own Role Visibility line explicitly lists "schedule changes for Student/Teacher," which is more specific and reflects the practical reality that a rescheduled Teacher needs to know too. Confirmed with the user; schedule-change notifications now go to both affected students and the entry's assigned Teacher.
- **Attendance-warning notification frequency (resolved during the Milestone 9 pre-implementation review, confirmed with the user):** dispatched only on a genuine threshold crossing (`>= 80%` immediately before the new attendance record, `< 80%` immediately after) тАФ not repeated on every subsequent marking while the student remains below 80%, satisfying Domain Rule 5's duplicate-prevention requirement without silently dropping the warning for a student who later crosses below the threshold again after recovering.
- **Exam-domain notifications (resolved during the Milestone 9 pre-implementation review, confirmed with the user):** the Milestone 9 kickoff's Domain Rules named "Exams" as a notification source, but no FR and no `notification.type` enum value covers exam-specific events. Resolved as: no new type or trigger added; `result_published` (fired from Milestone 7's `POST /results/{id}/approve`) is treated as the sole, sufficient notification for the exam-grading-to-result pipeline. Milestone 6's `exam.status = published` transition remains a UI-visibility change only, not a separate notification event.
- **Fee-due notification timing (resolved during the Milestone 9 pre-implementation review, confirmed with the user):** dispatched at invoice-issuance time (`POST /fees`'s auto-generation), not as a scheduled reminder before `due_date` тАФ no cron/scheduler mechanism exists anywhere in this project, and `BR-010` already leaves the exact reminder timing/frequency undefined. FR-044's "due-date reminder" half remains unimplemented for this reason (see the FR-044 row above).
- **FR-050**: No schedule-change-request endpoint is listed in the proposal's ┬з6 API spec.
